# 세션 인수인계 (2026-03-25)

## 이번 세션에서 확인한 목표
- 목표: Render 배포 대신, 사무실의 유휴 PC에서 Django 서버를 로컬로 구동.
- 운영 환경 후보: Windows 네이티브 / WSL2(Ubuntu).
- 결론: 사무실 상시 운영만 목적이면 Windows 네이티브 실행이 가장 단순.

## 중요한 결론
- 구글/네이버 API는 로컬 서버에서도 사용 가능.
- 로컬에서 안 되는 경우는 보통 아래 설정 이슈:
  - `.env` API 키/시크릿 누락 또는 오입력
  - OAuth 리디렉션 URI 미등록 (`127.0.0.1` 또는 내부 IP)
  - 사내망 아웃바운드 HTTPS 제한

## 현재 확인된 환경 이슈
- `python manage.py check` 실행 시 환경에 따라 패키지 누락이 발생할 수 있음:
  - `cloudinary_storage`
  - `google_auth_oauthlib`
- 해결: 반드시 프로젝트 전용 가상환경 활성화 후 `pip install -r requirements.txt`.

## 다음 세션 시작 체크리스트 (권장: Windows 네이티브)
1. 프로젝트 폴더 이동
   - `cd C:\Users\wonki\tshirt-management`
2. 가상환경 활성화
   - PowerShell: `.\venv\Scripts\Activate.ps1`
3. 패키지 설치/동기화
   - `pip install -r requirements.txt`
4. 환경파일 확인 (`.env`)
   - `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`
   - API 연동 시 Cloudinary/Google/Naver 관련 값 확인
5. DB 마이그레이션
   - `python manage.py migrate`
6. 점검
   - `python manage.py check`
7. 서버 실행
   - 개인 확인: `python manage.py runserver`
   - 같은 사무실 다른 PC 접속 필요 시: `python manage.py runserver 0.0.0.0:8000`

## 네트워크/사무실 공유 운영 메모
- 다른 PC에서 접속하려면:
  - `.env`의 `ALLOWED_HOSTS`에 서버 PC 내부 IP 추가
  - Windows 방화벽 인바운드 TCP 8000 허용
- WSL2로 운영 시 LAN 공유는 포트포워딩 이슈가 생길 수 있어 Windows 네이티브가 더 단순함.
